# Claude Code 扩展中文化 · 操作指南（SOP）

把 Cursor / VS Code 里的 **Claude Code 扩展**界面文字改成中文的完整原理与操作。

> ⚠️ **本文面向译表贡献者**——使用者请直接看 [README.md](README.md)（3 步上手就完事）。本文解决的是：怎么发现遗漏、怎么写新译表、怎么交叉审、扩展升级后怎么办等深度问题。
>
> ⚠️ **核心前提**：这是对官方扩展打包文件的「魔改」，**扩展一升级就会被覆盖、汉化全部失效**，需要重跑。改前务必备份（脚本会自动备份原版为 `index.js.orig`）。

## 〇·5、译表维护 SOP（5 阶段工作流）

> 本节解决"使用者的 3 步之后,译表怎么持续更新"。任何 Agent（Claude Code / Cursor 内置 / Codeium / ...）拿到本仓库都能按部就班执行。

按"扫描 → 翻译 → 交叉审 → 执行 → 复审"五阶段流水线跑。**注意：这套工作流是给译者（我们）用的，不是给使用者用的**——使用者只需 clone + 跑 `一键汉化.py`，跟他们没关系。

| 阶段 | 目的 | 主要动作 |
|---|---|---|
| **1. 扫描** | 认识战场 | 跑 `audit-命令汉化覆盖.py` + `audit-ui串缺口.py` |
| **2. 翻译** | 建清单 | 英文 → 中文，进 `*.json` 译表 |
| **3. 交叉审** | 质量关（必须不同模型） | 用 Codex / DeepSeek / 豆包 / Gemini 等做独立第二意见 |
| **4. 执行** | 重打汉化 | `python 一键汉化.py --audit` |
| **5. 复审** | 客观验收 | 复跑审计，命令覆盖率 ≥ 99% |

### 为什么要分阶段

- **扫描和复审是同一组脚本**——区别是扫描阶段是在干净基线上跑(发现缺口)，复审阶段是在新译表上跑(确认改善)
- **翻译阶段必须人工(或 + AI 辅助),不能全自动化**——跨语境术语需要判断(如 land 是"代码合入"不是"上线")
- **交叉审必须用不同模型**——同模型审自己会有盲点,用 Codex/Gemini/DeepSeek/豆包任选其一做独立第二意见

### 阶段间循环

第 5 步发现未达 99% → 回到第 2 步补译 → 再第 4 步 → 再第 5 步。**闭环比"一次性改对"重要**——本仓库的 99% 覆盖率就是 4 轮这样的迭代出来的。

## 一、对象与文件位置

- **扩展**：`anthropic.claude-code`
- **安装目录**（自动定位，无需手填）：
  - Cursor：`~/.cursor/extensions/anthropic.claude-code-<版本>-<平台>/`
  - VS Code：`~/.vscode/extensions/anthropic.claude-code-<版本>-<平台>/`
- **界面文字所在文件**：`webview/index.js`（约 4.8 MB，压缩打包，但界面文字是**干净的字符串字面量**，可安全替换）
- **CSS**：`webview/index.css`（一般不用动）

> 升级后版本号变、目录名也变。`一键汉化.py` 用 `glob` 自动取最新版，通常不必手动指定；要指定就用 `--ext <目录>`。

---

## 二、汉化范围：三类活，对症下药

界面文字不是铁板一块，按"在 `index.js` 里长什么样"分三类，处理方式完全不同：

| 类型 | 长什么样 | 怎么改 | 对应脚本 |
|---|---|---|---|
| **A. 结构化字面量** | `label:"Auto mode"`、`title:"..."` | 带前缀锚定替换 | `apply-锚定.cjs` |
| **B. 裸字面量 / JSX 文本节点** | `"Bad response"`、`{text:"You've come to..."}`、`Press [键] to ...` 的提示文字 | 精确字符串替换 | `apply-精确.cjs` |
| **C. 斜杠命令描述** | 命令菜单里**根本没渲染描述元素** | **改渲染**：补 span + 注入中文表 | `inject-斜杠命令说明.cjs` |

**为什么必须区分**：A/B 是"改数据"（替换现成的字符串）；C 是"改渲染"（原本这块 UI 压根不显示描述，光改数据没用，必须先让它渲染）。**先确认 UI 到底渲不渲染你要改的那块，再决定改数据还是改渲染**——这是本方法最大的认知点。

---

## 三、一键汉化.py 的 8 步流程

脚本按顺序做这些事（看懂它就看懂了整套 SOP）：

1. **定位扩展**：`~/.cursor` / `~/.vscode` 下找最新 `anthropic.claude-code-*`。
2. **备份原版（幂等）**：首次把 `index.js` 备份成 `index.js.orig`；之后每次都**从 `.orig` 还原再重打**，保证可重复、不叠层。
3. **精确替换**（`apply-精确.cjs` + `map1`）：替换裸字面量与 JSX 文本节点。
4. **锚定替换**（`apply-锚定.cjs` + `map2`）：替换 `label:`/`title:` 类，只动显示位、不误伤代码。
5. **UI 补漏**（`apply-ui补漏.py`）：处理设置/对话框/模板字符串（含 `${...}` 插值）等边角。
6. **注入命令描述**（`inject-斜杠命令说明.cjs` + `斜杠命令-中文说明.json`）：改渲染 + 注入中文表。
7. **注入 CLI spinner**（`apply-spinner.cjs`）：写 `~/.claude/settings.json`，187 动词 + 41 提示 + `language:"Chinese"`。只增不覆盖。
8. **语法校验**（`node --check`）：必须 `SYNTAX_OK`，防白屏。

`--audit` 会在最后追加跑两个审计（命令覆盖 + UI 串缺口）。

## 四、斜杠命令描述：为什么要"改渲染"+ 智能 ID 匹配

斜杠命令（`/xxx`）和普通界面文字不一样：

- **命令清单和英文描述是 `claude` CLI 二进制运行时下发的**，不在扩展文件里（在扩展里 grep 不到）。
- 命令菜单原本**只渲染命令名、不渲染描述**——所以必须**改渲染**：在命令名后补一个描述 `<span>`，数据来自 `斜杠命令-中文说明.json`。

### 关键坑：webview 的命令 ID ≠ 译表 key

webview 里命令是这样注册的：

```js
{id:"clear-conversation", label:"Clear conversation", description:"Start a new conversation"}
```

`id` 是 `clear-conversation`，但译表 key 是 CLI 短名 `clear`——**两边对不上，就会回退显示英文**。注入器内置了**智能匹配**来兜（按顺序尝试）：

1. `id` 直接查（去掉开头 `/`）
2. **剥离连字符 + 常见后缀**：`clear-conversation` → `clear`、`mcp-config` → `mcp`、`plan-mode` → `plan`（正则 `/[-_]?(conversation|config|command|mode|setup|level|help|control|alias)$/`）
3. **剥离前缀** `slash-command-`：`slash-command-help` → `help`
4. `label` 兜底
5. 英文 `description` 兜底（**绝不让面板空白**）

> ⚠️ 后缀剥离的正则**必须带上 `[-_]?` 把连字符一起吃掉**——只吃 `conversation` 会剩 `clear-`，照样匹配不上。这是个真踩过的坑。

少数 webview 专属命令（`new-conversation`/`attach-file`/`toggle-thinking` 等，CLI 里没有对应命令）在注入器里用 `WEBVIEW_EXTRAS` 字典直接手译兜底。

## 五、覆盖审计：取代"人工截图找英文"

> ⚠️ **不要靠"重载后肉眼滚菜单、截图报英文"**——命令近 200 个，必漏。用脚本全量比对。

```bash
python 资源脚本/audit-命令汉化覆盖.py
```

它从**三处真实来源**全量提取命令：

1. **CLI 二进制**：内置命令（5 种注册形态都覆盖：`type:"local|local-jsx|prompt"`、`menuDescription:`、名字在前/描述在前的静态串、`get description(){}` getter）。
2. **用户技能目录** `~/.claude/skills/`：扫 `SKILL.md` 前言。
3. **插件目录** `~/.claude/plugins/cache/`：第三方插件（superpowers、frontend-design 等）的技能。**← 这一处最容易漏，务必带上。**

输出：覆盖率 + 未汉化清单（含英文描述，可直接翻译）+ 写出 `缺口-待译.json`。闭环：**审计 → 翻译缺口 → 注入 → 复跑审计**，直到达标。

### UI 串缺口审计（补普通界面文字）

```bash
python 资源脚本/audit-ui串缺口.py <扩展>/webview/index.js
```

提取仍是英文的 `"键:字符串"` 字面量。**注意它的盲区**：只认 `label/title/placeholder/...` 这种**有 key 的**字面量；**匿名 JSX 文本节点（`{text:"..."}`、`Press [键] to ...`）没有 key，扫不到**，需要手动找（见下节）。

## 六、两个隐蔽盲区（本工具包专门踩过的坑）

### 6.1 匿名 JSX 文本节点

欢迎页/空状态的提示语，如：

- `What to do first? Ask about this codebase or we can start writing code.`
- `Make a CLAUDE.md file for instructions Claude will read every single time.`
- `You've come to the absolutely right place!`

它们是 `{text:"..."}` 形式的字面量，**可以替换**，但没有 `label:` 这种 key，结构化审计扫不到。找法：

```bash
# 在 index.js.orig 里直接找 {text:"大写开头的英文..."}
grep -oP '\{text:"[A-Z][^"]{3,200}"\}' <扩展>/webview/index.js.orig
```

找到后并入 `map1`，走精确替换。

### 6.2 动态键提示（`Press [Shift] [Tab] to ...`）

这类提示**键名是动态的**（Mac 显示 `Option`、Windows 显示 `Alt`，用户还能改快捷键），但很多人误以为整句不能动。真相是：

- **键名（`Shift`/`Tab`/`Ctrl`...）是 React 子元素**，由 React 单独渲染，**不参与字符串替换**。
- **后半句提示文本（`to automatically approve code edits`）是死的字符串字面量**，可安全替换。

所以只替换文本部分即可：`"to automatically approve code edits"` → `"可自动批准代码修改"`。键名照旧动态显示，**不会出现"键名变中文字"的事故**。

找法：

```bash
grep -oP 'className:[a-z]+\.key},"[^"]+"\)[^"]*\)"," ","[^"]{3,120}"' <扩展>/webview/index.js.orig
```

## 七、升级后怎么办

扩展升级后，旧 `index.js.orig` 对新版本无效，但**流程不变**：

```bash
python 一键汉化.py --audit   # 自动定位新版本目录、自动重新备份、重打全部汉化
# Ctrl+Shift+P → Developer: Reload Window
```

若新版本改了内部结构导致某些替换 0 命中或注入锚点失效，审计/注入日志会提示，按第四~六节排查。

## 八、故障排查

| 症状 | 原因 | 解法 |
|---|---|---|
| **面板白屏** | 替换改坏了 JS 语法 | `cp <扩展>/webview/index.js.orig <扩展>/webview/index.js` 还原后 Reload；再排查是哪条替换 |
| **`node --check` 失败** | 同上 | 同上；常见是译文里有未转义的引号 |
| **找不到 node** | PATH 里没 node | 装 node，或在 `一键汉化.py` 的 `find_node()` 里加你的路径 |
| **审计报"找不到 claude 二进制"** | claude 不在 PATH | `python audit-命令汉化覆盖.py --bin <claude路径>` 显式指定 |
| **注入日志 `ANCHOR_COUNT=0`** | 新版本改了渲染结构，锚点字符串变了 | 在新版 `index.js` 里找对应渲染段，更新 `inject` 脚本里的 `anchor` |
| **某命令仍是英文** | 译表没这条 / ID 没对上 | 跑覆盖审计找缺口；若审计说"已汉化"但仍英文，多半是 ID 匹配问题（见第四节） |
| **改完没生效** | 没重载 | `Ctrl+Shift+P` → `Developer: Reload Window`（会重启面板、打断当前会话） |
| **升级后全英文** | 升级覆盖了汉化 | 正常现象，重跑 `一键汉化.py` |

## 九、CLI 终端汉化（v0.2 新增）

扩展 webview 汉化只解决了"Cursor/VSCode 面板里看到的"，**CLI 终端里**的 spinner 词（"Incubating..."之类）、偶尔冒出来的提示语、还有 AI 的回复语言仍是英文。

Claude Code CLI 官方支持 3 个 `settings.json` 配置 key 来汉化这三块：

| Key | 作用 | 我们的值 |
|---|---|---|
| `language` | AI 回复语言 | `"Chinese"` |
| `spinnerVerbs` | spinner 词条（替换默认 30 个英文动词） | 187 条中文趣味动词 |
| `spinnerTipsOverride` | 提示语（替换默认英文 tip） | 41 条重译版 |
| `spinnerTipsEnabled` | 是否显示提示 | `true` |

> ⚠️ **关键设计**：`apply-spinner.cjs` 只增不覆盖——你已有的其他 key（model/permissions/env 等）一律保留，**备份**为 `settings.json.bak-<时间戳>`。卸载时 `--remove` 移除注入的 4 个 key，原文件结构完整还原。

### 用法

```bash
# 默认跑一键汉化时会自动调,不必单独跑
node 资源脚本/apply-spinner.cjs --merge

# 试运行(打印会做什么但不真写)
node 资源脚本/apply-spinner.cjs --dry-run

# 卸载(还原 settings.json)
node 资源脚本/apply-spinner.cjs --remove
```

> 思路来自 [KongBai1145/claude-code-zh-cn](https://github.com/KongBai1145/claude-code-zh-cn) @ MIT，感谢原作者。

### 词汇表来源

- **verbs（187 条）**：直接搬 KongBai1145 的纯趣味本地化（"光合作用中"、"蹦迪中"之类），纯本地化创意、无术语冲突
- **tips（41 条）**：重译 KongBai1145 的 31 条原文 + 补齐 10 条，遵守我们 `翻译清单.md` 术语表（plan mode → 计划模式、agent → 智能体 等）

### 为啥不用魔改 CLI 二进制

调研时看到 `duanmuyuan/claude-code-zh` 和 `Markjinli/claude-code-chinese` 走的是**改 CLI 二进制**路线（v2.1.113+ 后 CLI 编译成二进制，改起来非常痛）。我们故意不走那条路：
- **官方 key 永远受支持**，扩展/CLI 升级不会失效
- 缺点：能汉化的范围**限于官方暴露的 key**（spinner/tips/language），其他 CLI 内部硬编码英文改不到。但这三个 key 已经覆盖 90% 的 CLI 体验。

## 十、译文质量约定

见 [翻译清单.md](翻译清单.md)。要点：术语统一（response→回复、effort→思考力度、agent→智能体、session→会话）、纠真错译（如 land/landing 是"代码合入"不是"上线"）、中文全角标点、技术术语与代码标识符保留英文。
